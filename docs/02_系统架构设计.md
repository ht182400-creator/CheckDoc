# 02 系统架构设计

> 视角：架构师 + 全栈高级工程师

## 1. 架构总览

采用 **分层 + 配置驱动（Schema-Driven）** 架构，UI 与抽取逻辑通过 Schema 解耦，数据与代码通过 JSON 分离，保证可扩展与稳定。

```
┌──────────────────────────────────────────────────────────────┐
│                     UI 层 (NiceGUI)                           │
│  路径输入 · 扫描 · 热加载关键词 · 统计 · 筛选 · 表格 · 候选面板 · 导出 │
└──────────────┬───────────────────────────────────────────────┘
               │ 调用
┌──────────────▼──────────────────────────────────────────────┐
│                   编排/服务层                                  │
│  扫描 → 抽取 → 缓存 → 候选发现 → 渲染 流水线                    │
└──────────────┬──────────────────────────────────────────────┘
               │ 依赖
   ┌───────────┼────────────┬───────────────────┐
   ▼           ▼            ▼                   ▼
┌────────┐ ┌──────────┐ ┌───────┐  ┌──────────────────────┐
│Scanner │ │Extractor │ │ Store │  │ keywords_discovery.py│
│ 扫描   │ │ 抽取引擎  │ │ 缓存  │  │ 候选关键词发现引擎     │
└────────┘ └────┬─────┘ └───────┘  └──────────────────────┘
                │ 可选                       ▲ 查重
         ┌──────▼──────┐                     │
         │ LLM Client  │               ┌─────┴──────────┐
         └──────┬──────┘               │  keywords.py   │ ← 加载
                │                      │  热加载 / 添加  │
                ▼                      └──────┬─────────┘
         ┌──────────────┐                     │ 读写
         │  config.py   │ ← ← ← ─ ─ ─ ─ ─ ─ ┐
         │ Schema + 参数 │                     ▼
         └──────────────┘            keywords_data.json
                                       (纯 JSON 数据文件)
```

## 2. 模块职责

| 模块 | 职责 | 关键接口 |
| --- | --- | --- |
| `config.py` | 全局常量、字段 Schema（从 keywords.py 加载关键词） | `SCHEMA`, `LANG_OPTIONS`, `DEFAULT_ROOT` |
| `keywords.py` | 关键词加载器：JSON 读写 + `reload()` 热加载 + `add_keyword()` | `reload()`, `add_keyword()`, `get_all_known_keywords()` |
| `keywords_data.json` | 纯数据文件，存放全部关键词映射（不依赖代码） | 手工编辑或通过 UI 添加 |
| `keywords_discovery.py` | 候选发现引擎：扫描未覆盖术语，生成候选供确认 | `discover_from_text()`, `get_candidates()` |
| `logger.py` | 毫秒级日志 + 每日轮转，双输出 | `setup_logger()` |
| `scanner.py` | 递归发现 `memory` 目录与 MD 文件 | `scan_memory_md(root) -> List[FileMeta]` |
| `extractor.py` | 规则抽取 + LLM 兜底 + 候选发现 | `extract(file_meta) -> dict` |
| `llm_client.py` | OpenAI 兼容 HTTP 抽取（可选） | `LLMExtractor.extract(text, schema)` |
| `store.py` | 基于 mtime 的增量缓存 | `load_cache()`, `save_cache()`, `needs_update()` |
| `ui_app.py` | NiceGUI 界面：扫描/筛选/导出/热加载/候选面板 | `create_ui()`, `run_scan()` |
| `main.py` | 入口，装配并启动 | `main()` |

## 3. 关键设计决策

1. **Schema 驱动**：表格列、筛选控件、抽取目标全部由 `config.SCHEMA` 生成。新增字段 = 改配置。
2. **数据分离 + 热加载**：关键词从 `keywords_data.json` 加载，UI 按钮触发 `keywords.reload()` 热更新 config 模块全局变量，无需重启应用。
3. **抽取双引擎**：默认规则抽取（零依赖、离线可用）；可选 LLM 抽取（更准确，可插拔）。二者实现同一接口。
4. **候选发现**：当文件无关键词命中（language="通用"），自动从文本反引号/CamelCase/标题中提取未覆盖的技术术语，展示为候选列表。
5. **异常隔离**：单文件抽取失败 → 记日志 + 生成"解析失败"占位记录，不中断整体。
6. **增量缓存**：`store` 以 `path+mtime` 为键缓存抽取结果，重扫仅处理变更文件，满足性能 NFR。
7. **同步但异步友好**：扫描在 NiceGUI 协程内执行并刷新进度，避免长时间无响应。

## 4. 扩展性要点

- **加字段**：`config.SCHEMA` 追加 `FieldDef`，UI 自动出现新列与筛选器。
- **换抽取逻辑**：实现新的 `Extractor` 并在 `extractor.py` 注册。
- **换 LLM 厂商**：改 `LLM_BASE_URL` 即可（OpenAI 兼容协议）。
- **LLM 抽取 UI 化**（P1-2）：增加一键开关 + 批量重抽按钮 + 连接状态指示。
- **图谱视图**（P2-1）：新增 `graph_view.py` 模块，按语言×类型生成力导向聚类图。
- **加导出格式**：在 `ui_app.export_*` 增加分支。
- **定时同步**（P2-3）：新增 `sync.py` 模块，支持 cron 周期重扫 + diff 变更通知。

> 扩展路线详见 [08 综合评审报告](08_综合评审报告.md) 统一优先级总表。

## 5. 稳定性与性能

- 文件 IO / 网络（LLM）全部 try-catch，日志含 `traceback.format_exc()`。
- 大目录扫描使用生成器 + 进度回调，避免一次性载入内存。
- 缓存文件损坏时自动重建（catch JSONDecodeError）。
