## P6 rowClick 崩溃修复（2026-07-19 13:4x）
- 现象：扫描 2 万条后点击表格行，反复抛 `AttributeError: 'list' object has no attribute 'get'`（ui_app.py:26），导致详情打不开。
- 根因（已查 NiceGUI 源码确认，非猜测）：`nicegui/client.py:398-400` 在分发事件时，若 `msg['args']` 列表长度为 1 才解包成单值；而 `S.table.on("rowClick", _on_row_click)` **未指定 args**，Quasar 的 `rowClick` 发射 `(evt, row, pageIndex)`（`static/quasar.umd.js:23766` `emit("rowClick", evt, row, pageIndex)`），故 `e.args` 实为 3 元素列表 `[evt, row, pageIndex]`。旧 handler 直接 `e.args.get("row")`，对 list 调 `.get` 崩溃。
- 修复：
  - ui_app.py：`S.table.on("rowClick", _on_row_click, args=["row"])` 显式只取 row 对象（注册即解包）。
  - 抽纯函数 `_ui_helpers._parse_row_click_id(args)`：dict→取 `id`(int) 返回，其它（列表/None/无 id）返回 None（防御性，避免再崩）。

- 测试：tests/test_ui_helpers.py 新增 `TestParseRowClickId`（UH21-UH24）覆盖 row dict/无 id/列表/None，docs/09 同步（UIHelpers 22→26、Python 小计 82→86、总计 229→233）。

## IME 修复方案二次修正：composition 事件实测失效 → 改用 input+isComposing（2026-07-19 深夜）
- 实测：上一版 `widget.on("compositionstart"/"compositionend")` 设 `S.search_composing` **完全没生效**（归一化词仍是 `向dao`，无 "IME 合成状态变更" DEBUG）→ Quasar QInput **不把 compositionstart/end 转发到 Python 端**。
- 调研（Vue3 主流 + NiceGUI Generic Events 官方文档）：主流用原生 `input` 事件的 `InputEvent.isComposing` 标志——合成中=True 跳过、提交后=False 才搜，无需配对 composition 事件；NiceGUI `.on()` 官方确认支持任意原生 DOM 事件（含 `input`），通用事件参数默认带 DOM 属性，`isComposing` 可随 `input` 回传（显式只取 `["isComposing"]` 减负）；`input` 是高频原生事件 NiceGUI 必然支持，比 compositionstart 可靠。
- 最终方案（`_ui_filters._attach_ime_safe_input`）：原生 `input` 回调 `_on_input` 读 `isComposing`，True 跳过、否则防抖(SEARCH_DEBOUNCE_SEC=0.25)检索；防抖为**双保险**（即便 isComposing 取不到恒 False，停输后也必用中文检索，最终命中一定对）。新增 `_extract_is_composing(e)` 防御解析 args 多形态。
- 清理：删 `_ui_state.search_composing` 与 `_set_composing`（版本幽灵）；全局框/列 TEXT 筛选框均走此 helper。
- 覆盖：英文/数字/标识符(无合成,停输后搜)、中文经输入法(合成中跳过,提交后搜)、中英文混输、粘贴、清除(空词显示全部)。
- 验证：loop/verify.py ALL PASS；浏览器实测「向导」与 `xiangdao` 均应命中（若仍 0 命中，说明 `input` 的 isComposing 未回传，则降级纯防抖，再报）。

## IME 修复三次修正：input 事件不触发 Python 回调 → 主触发改回 update:model-value（2026-07-20 晚）
- 现象：用户反馈日志里**完全没有搜索记录**（`_compute_visible`/IME 跳过日志都不见），只有扫描相关日志。
- 根因：上一版把搜索主触发从 `update:model-value` 改成了 `widget.on("input", ...)`。但 Quasar QInput 内部走 `update:model-value`，**不把原生 `input` 事件作为 Python 回调触发**（实测整段搜索回调没跑）→ 日志无记录、搜索失灵。
- 修正（`_ui_filters._attach_ime_safe_input`）：主触发**改回已验证能触发的 `update:model-value`**（`_on_model_value` 读 composing_ref 决定是否跳过+防抖）；原生 `input` 事件降级为"尽力读 isComposing"的辅助（`_on_input` 仅写 composing_ref，用 try 包裹，绑定失败仅 WARNING 降级纯防抖）。这样搜索**一定触发**，IME 优化尽力而为。
- 关键教训（写死）：NiceGUI 的 `ui.input`(QInput) 只能可靠用 `update:model-value`/`on_change` 触发 Python 回调；原生 `input`/`compositionstart`/`compositionend` 等 DOM 事件**不一定转发到 Python 端**，不能用作主触发。今后类似"每键触发"需求一律用 `update:model-value`，原生 DOM 事件只作尽力辅助。
- 验证：loop/verify.py ALL PASS。
